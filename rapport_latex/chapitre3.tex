\chapter{Préparation, Planification et Architecture Système}

\section{Introduction}
Le succès d'un projet d'intelligence artificielle pour la smart city repose autant sur la gestion, l'organisation et la fiabilité du code que sur la performance des modèles. Ce chapitre détaille la logique de structuration choisie, les arguments derrière les choix techniques, et comment l'ensemble du projet Circuli est planifié pour assurer robustesse, flexibilité et évolutivité.

\section{Préparation Initiale}
Avant d'entamer le développement logiciel, une phase de préparation a permis de poser les bases techniques, identitaires et organisationnelles du projet.

\subsection{Environnement de développement et déploiement local (Ubuntu)}
Contrairement à un déploiement web classique nécessitant l'acquisition d'un nom de domaine via un hébergeur (de type OVH), nous avons privilégié une architecture locale déployée sur une machine fonctionnant sous \textbf{Ubuntu}. Ce choix technique s'inscrit dans une logique de "Edge Computing" : il permet de s'affranchir des latences réseau liées au Cloud et d'exploiter directement les ressources matérielles locales (CPU/RAM) pour assurer l'inférence du modèle d'Intelligence Artificielle en temps réel sur les flux vidéo.

Le projet implique trois groupes principaux :
\begin{itemize}

\section{Méthodologie et choix organisationnels}
Dès la conception, nous avons opté pour une organisation modulaire et industrialisée :
\begin{enumerate}
    \item \textbf{Découpage par domaines fonctionnels} : Deux modules principaux — \texttt{capture\_reconnaissance} (flux vidéo, IA) et \texttt{bi\_analytics} (API, KPIs, dashboard).
    \item \textbf{Conteneurisation systématique (Docker)} : Chaque composant est isolé, testable individuellement, redéployable sur n'importe quel serveur Ubuntu.
    \item \textbf{Workflow orchestré (Airflow)} : Toute la logique d'acquisition, détection, stockage, etc. est automatisée via des DAGs (Directed Acyclic Graphs).
    \item \textbf{Collaboration et DevOps} : Git/GitHub pour le code source, dashboarding partagé (Grafana), API documentée (FastAPI).
\end{enumerate}

\section{Détails techniques et justification des choix}
\subsection{Langage et IA}
Python 3.11+ a été choisi pour sa maturité en ML/Deep Learning et pour l'intégration avec YOLOv8, bibliothèque ultralytics leader pour la détection d'objets.

\subsection{API et orchestration}
\begin{itemize}
    \item \textbf{FastAPI} : Génère automatiquement la documentation interactive et permet un mapping simple entre endpoints et traitements IA/post-processing.
    \item \textbf{Apache Airflow} : Gère le lancement automatique/séquentiel des tâches critiques.
\end{itemize}

\subsection{Dashboard et stockage}
Le choix de Grafana et PostgreSQL permet de présenter les indicateurs clés de la mobilité en temps réel.

\subsection{Acquisition vidéo}
yt-dlp permet un streaming robuste de sources hétérogènes (YouTube live, RTSP, liens caméras municipales...) directement traitées par le serveur Ubuntu.

\subsection{Industrialisation}
La présence d'un script \texttt{setup\_all.sh} (exécuté sous bash Ubuntu) permet l'initialisation complète du projet, depuis le build des images Docker jusqu'au lancement de la stack complète via \texttt{docker-compose}.

\section{Conclusion}
Ce chapitre a permis de clarifier la méthodologie du projet.
