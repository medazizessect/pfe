\chapter{Préparation, Planification et Architecture Système}

\section{Introduction}
Le succès d'un projet d'intelligence artificielle pour la smart city repose autant sur la gestion, l'organisation et la fiabilité du code que sur la performance des modèles. Ce chapitre détaille la logique de structuration choisie, les arguments derrière les choix techniques, et comment l'ensemble du projet Circuli est planifié pour assurer robustesse, flexibilité et évolutivité.

\section{Méthodologie et choix organisationnels}
Dès la conception, nous avons opté pour une organisation modulaire et industrialisée :
\begin{enumerate}
    \item \textbf{Découpage par domaines fonctionnels} : Deux modules principaux — \texttt{capture\_reconnaissance} (flux vidéo, IA) et \texttt{bi\_analytics} (API, KPIs, dashboard).
    \item \textbf{Conteneurisation systématique (Docker)} : Chaque composant est isolé, testable individuellement, redéployable sur n'importe quel serveur/cloud.
    \item \textbf{Workflow orchestré (Airflow)} : Toute la logique d'acquisition, détection, stockage, etc. est automatisée via des DAGs (Directed Acyclic Graphs).
    \item \textbf{Collaboration et DevOps} : Git/GitHub pour le code source, dashboarding partagé (Grafana), API documentée (FastAPI).
\end{enumerate}

\section{Détails techniques et justification des choix}
\subsection{Langage et IA}
Python 3.11+ a été choisi pour sa maturité en ML/Deep Learning et pour l'intégration avec YOLOv8, bibliothèque ultralytics leader pour la détection d'objets.

\subsection{API et orchestration}
\begin{itemize}
    \item \textbf{FastAPI} : Génère automatiquement la documentation interactive et permet un mapping simple entre endpoints et traitements IA/post-processing.
    \item \textbf{Apache Airflow} : Gère le lancement automatique/séquentiel des tâches critiques telles que la récupération de flux, l'inférence IA, et l'export des résultats vers la base ou le dashboard.
\end{itemize}

\subsection{Dashboard et stockage}
Le choix de Grafana et PostgreSQL permet de présenter les indicateurs clés de la mobilité en temps réel, avec historisation, différents rôles d'accès (ingénieur, décideur) et portabilité des dashboards.

\subsection{Acquisition vidéo}
yt-dlp permet un streaming robuste de sources hétérogènes (YouTube live, RTSP, liens caméras municipales...) que ne supportent pas les librairies OpenCV natives.

\subsection{Industrialisation}
La présence d'un script \texttt{setup\_all.sh} permet l'initialisation complète du projet, depuis le build des images Docker jusqu'au lancement de la stack complète via \texttt{docker-compose} : aucune dépendance cachée, reproductibilité parfaite.

\section{Automatisation, CI/CD et reproductibilité}
Tout build, déploiement, reconfiguration est automatisé grâce au couplage Docker / Airflow :
\begin{itemize}
    \item Suppression du “it works on my machine”.
    \item Déploiement local ou cloud identique (tests, production).
    \item (À venir) : Intégration Github Actions ou GitLab CI pour les tests et la livraison continue.
\end{itemize}

\section{Sécurité, gestion des accès et scalabilité}
Gestion centralisée des secrets (credentials base/API dans variables d'environnement Docker), monitoring via Grafana, possibilité de répliquer le service capture sur d'autres flux autonomes.

\section{Architecture d'Acquisition : Du PoC à la Réalité Municipale}
Dans le cadre d'un déploiement réel en conditions de "Smart City", le système \textit{Circuli} est conçu pour s'interfacer directement avec le Système de Gestion Vidéo (VMS) de la municipalité (caméras IP CCTV via protocole RTSP). 

Toutefois, durant la phase de développement académique, l'accès direct aux serveurs sécurisés des municipalités est restreint. Pour surmonter cette contrainte, nous avons mis en place un \textbf{simulateur urbain} utilisant \texttt{yt-dlp} pour extraire les flux en direct de caméras urbaines diffusées sur YouTube, validant ainsi la chaîne d'ingestion algorithmique de manière iso-fonctionnelle.
